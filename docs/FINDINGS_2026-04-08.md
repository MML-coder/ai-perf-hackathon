## Root Causes Identified

### 1. open_file_cache disabled (CRITICAL)

**Original Config:**
```nginx
open_file_cache off;
```

**Problem**: With 2.5 million small files, nginx makes repeated `open()` syscalls for every request, creating massive overhead.

**Fix Applied:**
```nginx
open_file_cache max=10000 inactive=60s;
open_file_cache_valid 120s;
open_file_cache_min_uses 1;
open_file_cache_errors on;
```

**Result**: Small files improved from 383K to 1.28M rps (+234%)

### 2. worker_rlimit_nofile too low

**Original**: `worker_rlimit_nofile 1024`

**Problem**: Each worker could only open 1024 files, limiting concurrent connections and cached files.

**Fix**: `worker_rlimit_nofile 65535` + systemd LimitNOFILE=65535

### 3. worker_connections too low

**Original**: `worker_connections 1024`

**Fix**: `worker_connections 4096` with `use epoll` and `multi_accept on`

### 4. access_log overhead

**Original**: `access_log /var/log/nginx/access.log main;`

**Fix**: `access_log off;` (reduces I/O for benchmark)

## Failed Attempt: open_file_cache max=500000

**What happened**: 
- Set `open_file_cache max=500000` thinking bigger = better
- 112 workers × 500K = potentially 56 million file descriptors
- Nginx failed with "Too many open files" errors
- Performance dropped to 124K rps with 11,544 timeouts

**Lesson**: `open_file_cache max` must account for worker count:
```
safe_max = worker_rlimit_nofile / 2  # Leave room for connections
total_capacity = safe_max × num_workers
```

For 112 workers: max=10,000 → total 1.12M cached entries (safe)

## Medium File Analysis (Why No Improvement)

**Math:**
```
File size: 2 MB
Requests/sec: 1,400
Throughput: 2 MB × 1,400 = 2.8 GB/s = 22.4 Gbps
Network: 25 Gbps
Utilization: 89.6%
```

**Conclusion**: Medium files are **network-limited**, not nginx-limited. To improve:
- Upgrade to 100 Gbps network
- Enable compression (if files are compressible)
- This is NOT a misconfiguration

## Final nginx.conf

```nginx
user nginx;
worker_processes auto;
worker_rlimit_nofile 65535;
error_log /var/log/nginx/error.log;
pid /run/nginx.pid;

include /usr/share/nginx/modules/*.conf;

events {
    worker_connections 4096;
    use epoll;
    multi_accept on;
}

http {
    log_format main '$remote_addr - $remote_user [$time_local] "$request" '
                    '$status $body_bytes_sent "$http_referer" '
                    '"$http_user_agent" "$http_x_forwarded_for"';

    access_log off;
    
    sendfile on;
    tcp_nopush on;
    tcp_nodelay on;
    keepalive_timeout 65;
    keepalive_requests 10000;
    types_hash_max_size 4096;
    
    client_body_buffer_size 16k;
    client_max_body_size 10m;

    aio threads;
    directio 512k;

    open_file_cache max=10000 inactive=60s;
    open_file_cache_valid 120s;
    open_file_cache_min_uses 1;
    open_file_cache_errors on;
    
    include /etc/nginx/mime.types;
    default_type application/octet-stream;

    include /etc/nginx/conf.d/*.conf;
}
```

## System Tunings Applied

### Systemd Limits

```ini
# /etc/systemd/system/nginx.service.d/limits.conf
[Service]
LimitNOFILE=65535
```

### NVMe I/O

```bash
# I/O scheduler (lowest overhead)
echo 'none' > /sys/block/nvme0n1/queue/scheduler

# Read-ahead for 2MB files
blockdev --setra 8192 /dev/nvme0n1
```

## Key Takeaways

1. **open_file_cache is CRITICAL** for serving many small files
2. **Calculate FD limits carefully** with high worker counts
3. **Always check error.log** after config changes
4. **Medium/large file throughput** is often network-limited, not nginx-limited
5. **Benchmark before AND after** each change to isolate impact
6. **Start conservative**, increase gradually

## Rollback Commands

```bash
# Restore original nginx config
cp /etc/nginx/nginx.conf.backup.* /etc/nginx/nginx.conf
nginx -t && systemctl restart nginx

# Remove systemd override
rm /etc/systemd/system/nginx.service.d/limits.conf
systemctl daemon-reload
systemctl restart nginx

# Reset NVMe scheduler
echo 'mq-deadline' > /sys/block/nvme0n1/queue/scheduler
blockdev --setra 256 /dev/nvme0n1
```
