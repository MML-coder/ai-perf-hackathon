## Root Causes Identified

### 1. Wrong NIC After RHEL 9.7 Migration (PRIMARY)
- **Before**: Using `eno12399np0` (25 Gbps) at IP `10.1.89.236`
- **After**: Switched to `ens2f0np0.103` (100 Gbps) at IP `172.23.89.236`
- **Impact**: 4x bandwidth increase; medium/large/mixed files were network-limited at 25G
- **Evidence**: Medium baseline at 2.74 GB/s = ~22 Gbps = 87% of 25G NIC capacity

### 2. open_file_cache Disabled
- **Before**: `open_file_cache off`
- **After**: `open_file_cache max=10000 inactive=60s`
- **Impact**: Eliminates repeated file descriptor lookups for 2.5M small files

### 3. Low File Descriptor Limits
- **Before**: `worker_rlimit_nofile 1024`
- **After**: `worker_rlimit_nofile 65535` + systemd `LimitNOFILE=65535`
- **Impact**: Prevents FD exhaustion under high concurrency

### 4. Low worker_connections
- **Before**: `worker_connections 1024`
- **After**: `worker_connections 16384`
- **Impact**: Higher concurrent connection headroom per worker

### 5. Access Logging Enabled
- **Before**: `access_log /var/log/nginx/access.log main`
- **After**: `access_log off`
- **Impact**: At 1M+ rps, logging creates massive I/O overhead

### 6. Default TCP Stack
- **Before**: `cubic` congestion control, 212KB buffers, no busy polling
- **After**: `bbr`, 64MB buffers, `busy_poll=50`, `tcp_fastopen=3`
- **Impact**: Better throughput and lower latency for network-bound workloads

### 7. Suboptimal NIC Configuration
- **Before**: Ring buffers at 1024 (max 8192)
- **After**: Ring buffers at 8192, GRO/GSO/TSO enabled, adaptive coalescing, RPS
- **Impact**: Reduces packet drops and improves multi-core packet processing

### 8. No reuseport
- **Before**: `listen 80 default_server`
- **After**: `listen 80 default_server reuseport`
- **Impact**: Eliminates accept mutex contention across 112 workers

### 9. NVMe Scheduler Not Optimal
- **Before**: `mq-deadline`
- **After**: `none` (bypass scheduler for NVMe)
- **Impact**: Lower I/O latency for any non-cached reads

## Key Insight: Why NOT to Use directio

`directio 512k` was **not** applied in this run. Previous testing showed it causes ~50% degradation on medium/large/mixed workloads because:

- The DUT has **502GB RAM** and the dataset is **~489GB** — everything fits in page cache
- `directio` forces nginx to bypass page cache and read from disk for files >512KB
- `sendfile on` (without directio) serves files directly from page cache at memory speed
- Measured impact: medium files at 5,595 rps (without directio) vs 2,921 rps (with directio) = **91% better without**

## Tunings Applied

### Nginx (`/etc/nginx/nginx.conf`)
```
worker_processes auto;          # 112 workers
worker_rlimit_nofile 65535;
worker_connections 16384;
multi_accept on;
use epoll;
access_log off;
sendfile on;
tcp_nopush on;
tcp_nodelay on;
keepalive_requests 10000;
open_file_cache max=10000 inactive=60s;
open_file_cache_valid 30s;
open_file_cache_min_uses 2;
open_file_cache_errors on;
sendfile_max_chunk 2m;
output_buffers 4 2m;
```

### Nginx Server Block (`/etc/nginx/conf.d/hackathon.conf`)
```
listen 80 default_server reuseport;
```

### Systemd (`/etc/systemd/system/nginx.service.d/limits.conf`)
```
[Service]
LimitNOFILE=65535
```

### Kernel TCP
```
net.core.somaxconn = 65535
net.core.rmem_max = 67108864
net.core.wmem_max = 67108864
net.ipv4.tcp_rmem = 4096 1048576 67108864
net.ipv4.tcp_wmem = 4096 1048576 67108864
net.ipv4.tcp_congestion_control = bbr
net.core.netdev_max_backlog = 65535
net.core.busy_poll = 50
net.ipv4.tcp_fastopen = 3
vm.swappiness = 1
```

### NVMe
```
scheduler: none
read_ahead_kb: 8192
```

### NIC (ens2f0np0 - 100G)
```
ring buffers: rx 8192 tx 8192
offloads: gro on, gso on, tso on
coalescing: adaptive-rx on, adaptive-tx on
RPS: enabled across all CPUs
```
